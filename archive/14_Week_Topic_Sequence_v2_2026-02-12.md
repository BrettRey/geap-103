# GEAP 103: 14-Week Topic Sequence (Draft v2)

**Course:** GEAP 103 Basic Computer Skills (Foundation Semester, B1)
**Version:** Draft v2, 2026-02-12
**Status:** Revised after simulated expert review (Ellis, Wilson, Mayer perspectives)
**Arc:** Discovery (Weeks 1-4) → Building (Weeks 5-10) → Creation (Weeks 11-14)
**Contact time:** 3 hours/week (42 hours total)
**Environment:** Office 365 + Copilot, BYOD

## Design Principles

1. **One new tool or concept per week.** Students need time to practise before anything is assessed.
2. **Recurring task types build fluency.** The same task types (articulate goal, evaluate output, write verification note) recur across different content so that the language becomes automatic.
3. **Assessment weeks consolidate; they do not introduce.** Weeks 4, 8, 12 are check-in weeks. No new tools.
4. **Technical content serves language development.** Every tool or concept is a context for meaningful input and output in English. Material that does not serve the dual objective is excluded (coherence principle).
5. **Scaffolded freedom.** Early weeks are tightly structured; later weeks give students increasing choice.
6. **Worked examples before independent performance.** Every new tool-task combination begins with instructor demonstration or class-wide worked example. Guidance fades across subsequent weeks.
7. **Vocabulary pre-training.** New terms are introduced at the start of each session, before the tool interaction, not during it.
8. **Pair work throughout.** Structured pair tasks appear every week, not only in the collaboration phase. Interaction is a condition for language acquisition, not a Phase 3 add-on.
9. **Deliberate mismatch.** AI interaction tasks are designed so that AI output sometimes fails to match student intent, requiring genuine meaning negotiation in the review stage.

## Session Structure (Standard 3-Hour Block)

Each 3-hour class follows a predictable three-segment structure:

| Segment | Duration | Purpose |
|---------|----------|---------|
| **A: Vocabulary + Worked Example** | ~40 min | Pre-train 3-5 new terms. Instructor demonstrates the week's tool-task combination as a worked example. Students observe, ask questions. |
| **B: Guided Practice** | ~70 min | Students attempt the task with instructor support. Pair work integrated. Decision Log entry or verification note written. |
| *Break* | ~10 min | |
| **C: Independent/Pair Application + Wrap-up** | ~60 min | Students apply the skill to their own content or a partner's. Brief whole-class debrief (5-10 min): one student shares, class discusses. |

Assessment weeks (4, 8, 12) replace this structure with diagnostic + micro-defences + peer feedback.

## Phase 1: Discovery (Weeks 1-4)

### Week 1: Where Do Files Live?

**Focus:** File-system mental model; from app-centric to file-centric thinking

**New vocabulary (5 terms):** folder, subfolder, file path, file extension, cloud sync

**Segment A -- Worked example:** Instructor shows their own file system (OneDrive). Walks through: "Here is a folder. Here is a file inside it. Here is the path. Here is what happens when I sync." Students observe and identify vocabulary.

**Segment B -- Guided practice:** Students open their own OneDrive. Map their current file system on paper (diagnostic). In pairs, compare maps: "Where are your photos? Where are your school files? How is yours different from mine?"

**Segment C -- Application:** Create a course folder structure (using a template). Name 3 files following naming conventions. In pairs, review each other's folder structure: "Can you find [file]?"

**CLOs activated:** 3 (begin)
**Pair work:** Paired file-system audit; partner review of folder structure

---

### Week 2: First AI Interactions + Decision Logging

**Focus:** First use of Copilot in Word; introducing the Decision Log as a worked example

**New vocabulary (5 terms):** prompt, generate, revise, accept/reject, verification

**Segment A -- Worked example:** Instructor uses Copilot to draft a short document (e.g., class directory entry), deliberately showing a case where the AI output doesn't quite match the goal. Instructor completes a Decision Log entry on the projector, thinking aloud: "My goal was X, the AI gave me Y, I decided to change Z because..."

**Segment B -- Guided practice:** Students use Copilot to draft their own class directory entry. Complete first Decision Log entry (structured template, class does it together). In pairs, compare AI outputs: "What did yours say? What did you change?"

**Segment C -- Application:** Second AI task (e.g., "Use Copilot to summarise this paragraph" -- instructor provides a paragraph with a deliberate problem the AI is likely to miss). Students write a second Decision Log entry independently. Partners exchange and discuss.

**CLOs activated:** 1 (begin), 4 (begin)
**Assessment:** Decision Log entries begin
**Pair work:** Comparing AI outputs; exchanging Decision Log entries

---

### Week 3: Deepening File Management + Copilot in Practice

**Focus:** Consolidation of Weeks 1-2 through varied repetition; file formats and version awareness

**New vocabulary (5 terms):** .docx, .pdf, .xlsx, version history, export

**Segment A -- Worked example:** Instructor demonstrates file format differences: open same content as .docx, .pdf, .txt. Shows OneDrive version history: "I changed this file yesterday. Here is the old version. Here is the new version." Demonstrates export to PDF.

**Segment B -- Guided practice:** Students take their Week 2 document and save it in three formats. Use OneDrive version history to find the earlier version. In pairs, troubleshoot: "My PDF doesn't look right -- why?" (First informal troubleshooting practice, scaffolded by pair interaction.)

**Segment C -- Application:** Use Copilot for a new task (e.g., formatting a provided text as a professional email). Decision Log entry. Instructor provides an AI output with a deliberate formatting error; students identify it in pairs.

**CLOs activated:** 3, 1, 4
**Pair work:** Paired troubleshooting; collaborative error identification in AI output
**Note:** This week is a deliberate consolidation week. No new tool is introduced. The "one new concept" is file formats/version history, which extends the Week 1 file-system model.

---

### Week 4: Check-in 1

**Focus:** First assessment checkpoint. No new tools introduced.

**Activities:**
- In-class written diagnostic (10 min, unmediated): "Describe how you organised your files for this course and why."
- Oral Micro-Defence 1 (while others work on portfolio organisation)
- Peer feedback on walkthroughs

**Assessment:**
- File System artefact due (7%)
- Oral Micro-Defence 1 (incl. diagnostic + peer feedback)

**CLOs assessed:**
- CLO 3: evaluative (file system artefact)
- CLOs 5, 6: **diagnostic only** (baseline for comparison with Weeks 8 and 12; not included in grade calculation for these CLOs)

## Phase 2: Building (Weeks 5-10)

### Week 5: Version Control Concepts + First Git Steps

**Focus:** Why version control? Git as a timeline of snapshots.

**New vocabulary (5 terms):** repository, commit, snapshot, version history (Git sense), restore

**Segment A -- Visual pre-training + worked example:** Before touching any tool, instructor presents a visual model of version control: an animated or drawn timeline showing snapshots accumulating. "You already know OneDrive version history (Week 3). Git is the same idea, but more powerful." Then, worked example: instructor creates a repository, adds a file, makes a commit, changes the file, makes another commit, and restores the first version. Students observe.

**Segment B -- Guided practice:** Students create a repository for their course portfolio. Add their Week 1 file map and Week 2 document. Make first commits (AI-assisted: "Copilot, how do I commit this file?"). Change a file and commit again. In pairs, one student changes a file, partner talks them through committing.

**Segment C -- Application:** Each student practises the Restore episode: change a file, commit, then restore the previous version. Write a Decision Log entry about the experience. Pairs compare: "What happened when you restored?"

**CLOs activated:** 2 (begin), 3
**Pair work:** Partner-guided commits; paired restore practice

---

### Week 6: Documents with AI + Accessibility

**Focus:** Creating and revising documents with Copilot; accessibility fundamentals

**New vocabulary (5 terms):** heading (style), alt text, screen reader, Accessibility Checker, contrast

**Segment A -- Worked example:** Instructor shows two versions of the same document: one with no headings/alt text (just bold text and images), one properly structured. Runs Accessibility Checker on both. "What did the checker find? Why does this matter?" Then demonstrates fixing one issue with Copilot.

**Segment B -- Guided practice:** Students take a poorly formatted document (provided). Use Copilot to restructure it (add headings, alt text). Run Accessibility Checker. Write a verification note in their own words: what AI did, what they changed, what they checked.

**Segment C -- Application:** Apply accessibility principles to their own course document (from Week 2 or 3). Pairs review each other's documents with Accessibility Checker and give feedback.

**CLOs activated:** 7 (begin), 4
**Assessment:** Decision Log feedback round 1 returned; Git episodes begin (Restore opportunity from Week 5)
**Pair work:** Peer accessibility review

---

### Week 7: Spreadsheets with AI (Part 1)

**Focus:** Working with structured data in Excel + Copilot

**New vocabulary (5 terms):** cell, row, column, formula, chart

**Segment A -- Worked example:** Instructor demonstrates Excel + Copilot with a small dataset (e.g., class survey results). Shows: entering data, asking Copilot to create a formula, asking Copilot to create a chart. Then demonstrates the "can you explain this?" check: "Copilot, what does this formula do?" Verifies by checking one calculation manually.

**Segment B -- Guided practice:** Students work with a provided dataset. Use Copilot to create a summary table and chart. Verify one formula manually. Write Decision Log entry.

**Segment C -- Application:** Pairs work with a different dataset. One student creates a chart; partner evaluates it: "Does this chart show what you wanted? Is anything misleading?"

**CLOs activated:** 7, 1, 4
**Pair work:** Partner evaluation of charts

---

### Week 8: Check-in 2 + Troubleshooting

**Focus:** Second assessment checkpoint + explicit troubleshooting instruction

**New vocabulary (5 terms):** error message, troubleshoot, debug, help request, log

**This week combines an assessment check-in with the introduction of troubleshooting, making it a denser week. The troubleshooting instruction is integrated into the session rather than deferred to a standalone week.**

**Session structure (modified):**

**Segment A (40 min) -- Troubleshooting worked example:** Instructor presents 3 real error messages (from Office 365, Git, or Copilot). Walks through the parsing structure: "What happened? Where? What should I try?" Demonstrates writing a help request (the genre: context → problem → what I tried → what I expect).

**Segment B (70 min) -- Micro-defences:** In-class written diagnostic (10 min). Oral Micro-Defence 2 runs while other students practise troubleshooting: given 2 error messages, interpret them and write a help request for one.

**Segment C (50 min) -- Troubleshooting practice:** In pairs, students troubleshoot a staged problem (e.g., "This Excel formula gives the wrong answer -- why?"). Write help request if stuck. Decision Log entry about the troubleshooting experience.

**Assessment:**
- Oral Micro-Defence 2 (incl. diagnostic + peer feedback)

**CLOs assessed:** 5, 6 (now both taught and assessed)

---

### Week 9: Collaboration Concepts + PRs/Issues

**Focus:** Asynchronous collaboration through GitHub; pull requests and issues as professional communication

**New vocabulary (5 terms):** pull request (PR), issue, fork, comment, review

**Segment A -- Worked example:** Instructor demonstrates the full PR workflow: fork a repository, make a change, write a PR description, receive a comment, respond. Then demonstrates writing a GitHub Issue. Shows models of good and bad PR descriptions ("This one says what changed and why; this one just says 'fixed stuff'").

**Segment B -- Guided practice:** In pairs, students fork a shared practice repository, make a small change (e.g., add their name to a list), submit a PR with a description, and comment on their partner's PR. Instructor circulates.

**Segment C -- Application:** Write a GitHub Issue for a real or hypothetical problem in the shared repository. Partners respond to each other's issues. Decision Log entry about collaboration experience.

**CLOs activated:** 8 (begin), 6, 2
**Pair work:** Paired PR exercise; issue response
**Note:** This is the collaboration introduction. Students have 5 weeks (9-14) before collaborative work is assessed in the final project, vs. 4 weeks in v1.

---

### Week 10: Mid-Semester Project + Presentations

**Focus:** Submitting mid-semester artefact; practical slide creation with accessibility

**New vocabulary (5 terms):** slide, layout, speaker notes, caption, colour contrast

**Segment A -- Worked example:** Instructor creates a 3-slide presentation using Copilot. Runs Accessibility Checker. Shows: alt text on images, sufficient contrast, meaningful slide titles. (Visual communication theory dropped per Wilson; focus on accessible, functional slides.)

**Segment B (split):**
- First 30 min: Students finalise and submit mid-semester artefact with verification note.
- Remaining 40 min: Students create a 3-5 slide presentation on a topic of choice using Copilot. Run Accessibility Checker.

**Segment C -- Application:** Partners review each other's slides for accessibility. Write verification note.

**CLOs activated:** 7, 1, 4
**Assessment:** Mid-Semester Artefact due (8%); Decision Log feedback round 2 returned
**Pair work:** Peer accessibility review of slides

---

## Phase 3: Creation (Weeks 11-14)

### Week 11: Collaborative Project Launch

**Focus:** Defining collaborative projects; working in shared repositories

**No new vocabulary introduced.** This week consolidates terms from Weeks 5, 9, and earlier.

**Segment A -- Project setup:** Instructor presents project options and constraints. Students form pairs or small groups. Groups define project scope (what will they build?), divide work, and create a shared repository. Instructor models a project kick-off: "Here is my goal, here is how I'll divide the work, here is my first issue."

**Segment B -- Project work:** Groups begin building. Each student works on their portion. Instructor circulates for troubleshooting. Each student must submit at least one PR with description (Collaboration episode for Git Episodes).

**Segment C -- Status check:** Each group gives a 2-minute status update (informal, not graded): "What we're building, what's done, what's next." Class asks questions. This practises the status-update genre.

**CLOs activated:** 8, 6, 2
**Pair work:** Collaborative project work throughout

---

### Week 12: Check-in 3 + Project Continuation

**Focus:** Final assessment checkpoint and continued project work

**No new vocabulary introduced.**

**Session structure (modified):**

**Segment A (40 min) -- Portfolio curation preview:** Instructor demonstrates how to map evidence to CLOs for the reflection. Shows a model Evidence-Indexed Reflection entry (worked example). Students identify their own strongest evidence for 2-3 CLOs.

**Segment B (70 min) -- Micro-defences + project work:** In-class written diagnostic (10 min). Oral Micro-Defence 3 runs while other students continue project work.

**Segment C (50 min) -- Project work:** Continued collaborative work. Instructor available for troubleshooting.

**Assessment:**
- Oral Micro-Defence 3 (incl. diagnostic + peer feedback)

**CLOs assessed:** 5, 6

---

### Week 13: Project Refinement + Portfolio Curation

**Focus:** Iterating on projects; curating portfolio evidence for reflection

**No new vocabulary introduced.**

**Segment A -- Self-assessment calibration:** Instructor returns Decision Log self-assessment patterns (where student ratings diverged from instructor ratings). Brief discussion: "What does 'Proficient' look like? How do you know?" Worked example of an Evidence-Indexed Reflection entry, including the evaluative judgment components ("How strong is this evidence?" "What would I still need to work on?").

**Segment B -- Portfolio curation workshop:** Students map their evidence to CLOs and draft reflection notes. Instructor available for individual consultation. Partners exchange reflections and give feedback: "Is this evidence convincing? What's missing?"

**Segment C -- Project refinement:** Final iteration on collaborative projects. Decision Logs due (8 entries, 30%).

**CLOs activated:** All
**Assessment:** Decision Logs due
**Pair work:** Peer review of reflection drafts

---

### Week 14: Final Submissions + Course Wrap-up

**Focus:** Final artefact submission, reflection, and course conclusion

**Segment A -- Final submissions:** Students submit final artefact (with verification note), Git Episodes, and Evidence-Indexed Reflection. Instructor available for last-minute questions.

**Segment B -- Sharing:** Each student gives a 2-3 minute informal presentation: show one thing you built, say one thing you learned. Not graded. Celebratory.

**Segment C -- Course wrap-up:** "What can you do now that you couldn't in Week 1?" Discussion of transfer: how these skills apply in degree programs. Course evaluation.

**Assessment:**
- Final Artefact due (10%)
- Git Episodes due (10%)
- Evidence-Indexed Reflection due (10%)

## Fluency Development Strategy (Nation's Four Strands)

| Strand | Where it happens | Recurrence |
|--------|-----------------|------------|
| **Meaning-focused input** | Reading AI output, error messages, technical instructions | Every week |
| **Meaning-focused output** | Decision Logs, verification notes, help requests, PR descriptions | Every week |
| **Language-focused learning** | Vocabulary pre-training at start of each session; genre templates (help request, verification note, PR description) | Every week |
| **Fluency development** | Repeated task types with varying content: Decision Logs (8x same template), Micro-Defences (3x same oral genre), verification notes (3x same structure), AI interaction cycle (12 weeks of articulate → review → decide) | Built into structure |

**Key revision from v1:** The AI interaction cycle is now designed with deliberate intention-output mismatch in early weeks (Weeks 2-5), so that the review stage requires genuine meaning negotiation. In later weeks, mismatch decreases as students develop more precise goal articulation, but the AI still produces imperfect output often enough to maintain the negotiation demand.
