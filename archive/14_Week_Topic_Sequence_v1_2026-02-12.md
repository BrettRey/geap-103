# GEAP 103: 14-Week Topic Sequence (Draft v1)

**Course:** GEAP 103 Basic Computer Skills (Foundation Semester, B1)
**Version:** Draft v1, 2026-02-12
**Arc:** Discovery (Weeks 1-4) → Building (Weeks 5-10) → Creation (Weeks 11-14)
**Contact time:** 3 hours/week (42 hours total)
**Environment:** Office 365 + Copilot, BYOD

---

## Design Principles

1. **One new tool or concept per week.** Students need time to practise before anything is assessed.
2. **Recurring task types build fluency.** The same kinds of tasks (articulate a goal, evaluate output, write a verification note) recur across different content so that the language becomes automatic.
3. **Assessment weeks consolidate; they do not introduce.** Weeks 4, 8, 12 are check-in weeks with micro-defences. No new tools are introduced.
4. **Technical content serves language development.** Every tool or concept is a context for meaningful input and output in English.
5. **Scaffolded freedom.** Early weeks are tightly structured; later weeks give students increasing choice over what they build.

## Phase 1: Discovery (Weeks 1-4)

### Week 1: What Are "Basic Computer Skills" in 2026?

**Focus:** Course orientation; mental model shift from app-centric to file-centric

**Topics:**
- Course introduction: this is not a typing class; the "basic skill" is managing AI as a teammate
- Where do files live? The app-centric mental model (files live "in" Google Docs, "in" Photos) vs. the file-system model (files live on disk, in folders, synced to cloud)
- OneDrive tour: local folders, cloud sync, sharing, version history
- File types and extensions: why .docx ≠ .pdf ≠ .txt matters

**In-class activity:** Map your current file system. Where are your files right now? Draw it. (Low-stakes diagnostic of students' starting mental models.)

**Language focus:** Vocabulary for location and structure (folder, subfolder, path, root, extension, sync, local, cloud)

**CLOs activated:** 3 (begin)

---

### Week 2: First AI Interactions + Decision Logging

**Focus:** First substantive use of Copilot; introducing the Decision Log

**Topics:**
- Copilot in Word: ask it to generate, revise, summarise
- The cycle: articulate goal → review output → accept/modify/reject
- Introducing the Decision Log template: how to document what you asked, what you got, and what you decided
- Naming conventions: why "Essay final FINAL v2 (1).docx" is a problem, and what to do instead

**In-class activity:** Use Copilot to draft a short document (e.g., a self-introduction for a class directory). Complete first Decision Log entry together as a class.

**Language focus:** Task articulation language ("I need...", "Can you...", "Change this to..."); decision language ("I accepted this because...", "I changed... because...")

**CLOs activated:** 1 (begin), 4 (begin)
**Assessment:** Decision Log entries begin

---

### Week 3: Version Control Concepts + First Git Steps

**Focus:** Why version control? Git as a timeline of snapshots.

**Topics:**
- The problem version control solves: "I deleted my paragraph", "Which version did I send?", "Who changed what?"
- Mental model: a commit is a snapshot; your project has a timeline you can travel along
- First repository: create, add files, commit (with Copilot assistance)
- GitHub Desktop or CLI basics (AI-assisted)
- Connecting Git to file organisation: Git builds on the folder/file understanding from Weeks 1-2

**In-class activity:** Create a repository for your course portfolio. Add your Week 1 file map and Week 2 document. Make your first commits. Practise changing a file and committing again.

**Language focus:** Version control vocabulary (repository, commit, snapshot, version, restore, history); temporal language ("before", "after", "previous version", "current state")

**CLOs activated:** 2 (begin), 3

---

### Week 4: Check-in 1

**Focus:** Consolidation and first assessment checkpoint. No new tools introduced.

**Topics:**
- File system artefact preparation and submission
- Review: file organisation, naming, AI interaction, Decision Logs so far
- Oral micro-defence practice: what does a good technical walkthrough look like?

**In-class activities:**
- In-class written diagnostic (10 min, unmediated)
- Oral Micro-Defence 1 (while others work on portfolio organisation)
- Peer feedback on walkthroughs

**Assessment:**
- File System artefact due (7%)
- Oral Micro-Defence 1 (incl. diagnostic + peer feedback)

**CLOs assessed:** 3, 5, 6

---

## Phase 2: Building (Weeks 5-10)

### Week 5: Documents with AI + Accessibility

**Focus:** Creating and revising documents with Copilot; introduction to accessibility

**Topics:**
- Word + Copilot: drafting, restructuring, formatting
- Accessibility basics: headings and styles (not just bold text), alt text for images
- The Accessibility Checker: what it finds, what it means, what to do
- Verification notes: documenting what AI did vs. what you did (in your own words)

**In-class activity:** Take a poorly formatted document (provided). Use Copilot to improve it. Run Accessibility Checker. Write a verification note.

**Language focus:** Accessibility vocabulary (heading, alt text, screen reader, contrast, caption); instruction language for verification notes

**CLOs activated:** 7 (begin), 4

---

### Week 6: Spreadsheets with AI

**Focus:** Working with structured data in Excel + Copilot

**Topics:**
- Excel basics: cells, rows, columns, sheets (many students have limited spreadsheet experience)
- Copilot in Excel: creating formulas, charts, summaries
- Understanding what a formula does (even if AI wrote it): the "can you explain this?" check
- Git episodes begin: opportunity for Restore episode (recover a previous version of a file)

**In-class activity:** Given a small dataset (e.g., class survey results), use Copilot to create a summary table and chart. Verify one formula by checking it manually.

**Language focus:** Data vocabulary (row, column, cell, range, formula, chart, axis, label); causal language for explaining formulas ("This adds up... because...")

**CLOs activated:** 7, 1, 4
**Assessment:** Decision Log feedback round 1 returned; Git episodes begin

---

### Week 7: Reading Technical Text + Troubleshooting

**Focus:** Interpreting error messages, instructions, and technical documentation

**Topics:**
- What error messages actually say: parsing the structure (what happened, where, what to try)
- Troubleshooting workflow: read → try something → document what happened → ask for help
- Writing help requests: the genre (context, what I tried, what happened, what I expected)
- Technical vocabulary: a curated list of 10-15 essential terms for the course

**In-class activity:** Instructor provides 3-4 real error messages from Office 365 / Git. Students interpret each one, try a fix, and write a help request for one they can't solve.

**Language focus:** Help request genre moves (context → problem → what I tried → what I need); interpreting hedged/technical language in error messages

**CLOs activated:** 5 (primary focus), 6 (begin)

---

### Week 8: Check-in 2

**Focus:** Mid-semester consolidation and second assessment checkpoint. No new tools introduced.

**Topics:**
- Review: document creation, spreadsheets, troubleshooting, Decision Logs
- Mid-semester project planning: what will you build for Week 10?
- Error recovery episode opportunity (for Git Episodes)

**In-class activities:**
- In-class written diagnostic (10 min, unmediated)
- Oral Micro-Defence 2 (while others work on projects)
- Peer feedback on walkthroughs

**Assessment:**
- Oral Micro-Defence 2 (incl. diagnostic + peer feedback)

**CLOs assessed:** 5, 6

---

### Week 9: Presentations with AI + Visual Communication

**Focus:** Slides and visual communication with Copilot

**Topics:**
- PowerPoint + Copilot: generating slides, speaker notes, design suggestions
- Visual communication principles: one idea per slide, readable fonts, meaningful images
- Accessibility in presentations: alt text on images, sufficient colour contrast, captions
- Connecting to mid-semester project: could your project be a presentation?

**In-class activity:** Create a 5-slide presentation on a topic of choice using Copilot. Run Accessibility Checker. Revise based on results.

**Language focus:** Presentation vocabulary (slide, layout, speaker notes, transition); evaluative language for design choices ("I chose this because...", "This is clearer than...")

**CLOs activated:** 7, 1, 4

---

### Week 10: Mid-Semester Project Due + Collaboration Introduction

**Focus:** Submitting mid-semester artefact; introducing collaborative workflows

**Topics:**
- Mid-semester artefact finalisation and verification note
- Introduction to collaboration: GitHub as asynchronous communication
- Pull requests: what they are, why they exist, how to write a description
- Issues: reporting problems, requesting changes, discussing plans
- Preview of collaborative project (Weeks 11-14)

**In-class activity:** Submit mid-semester artefact. Then: in pairs, fork a shared practice repository, make a change, submit a PR with a description, and comment on your partner's PR.

**Language focus:** Collaborative communication genres (PR description, issue, comment); polite professional tone in async communication

**CLOs activated:** 8 (begin), 6, 2
**Assessment:** Mid-Semester Artefact due (8%); Decision Log feedback round 2 returned

---

## Phase 3: Creation (Weeks 11-14)

### Week 11: Collaborative Project Work

**Focus:** Sustained collaborative project in pairs or small groups

**Topics:**
- Project scoping: defining what to build, dividing work, setting milestones
- Working in a shared repository: who does what, branching if appropriate
- PR descriptions and code review comments as professional writing
- Bug reports and collaboration notes: documenting what went wrong and what was decided

**In-class activity:** Project work in pairs/groups. Instructor circulates for troubleshooting. Each student must submit at least one PR with description and one comment on a partner's PR (Collaboration episode for Git Episodes).

**Language focus:** Collaboration language (task delegation, status updates, feedback); written genres: bug report, collaboration note, handoff documentation

**CLOs activated:** 8, 6, 2

---

### Week 12: Check-in 3 + Project Continuation

**Focus:** Final assessment checkpoint and continued project work. No new tools introduced.

**Topics:**
- Project progress check: what's done, what's left, what problems have emerged
- Troubleshooting log review: what have you encountered and how did you handle it?
- Portfolio curation preview: what evidence do you have for each CLO?

**In-class activities:**
- In-class written diagnostic (10 min, unmediated)
- Oral Micro-Defence 3 (while others continue project work)
- Peer feedback on walkthroughs

**Assessment:**
- Oral Micro-Defence 3 (incl. diagnostic + peer feedback)

**CLOs assessed:** 5, 6

---

### Week 13: Project Refinement + Portfolio Curation

**Focus:** Iterating on projects; curating portfolio evidence for reflection

**Topics:**
- Iteration: revisiting and improving your artefact based on feedback and testing
- Self-assessment calibration: reviewing your Decision Log self-ratings against instructor feedback
- Portfolio curation: for each CLO, identifying your strongest evidence
- Preparing the Evidence-Indexed Reflection: the format, what "evaluating your evidence" means

**In-class activity:** Decision Logs due. Portfolio curation workshop: students map their evidence to CLOs and draft reflection notes. Instructor available for individual consultation.

**Language focus:** Evaluative language for self-assessment ("My strongest evidence is...", "This shows... because...", "I still need to work on...")

**CLOs activated:** All
**Assessment:** Decision Logs due (8 entries, 30%)

---

### Week 14: Final Submissions + Course Wrap-up

**Focus:** Final artefact submission, reflection, and course conclusion

**Topics:**
- Final artefact polishing and verification note
- Git Episodes compilation: ensure all 3 episodes are documented
- Evidence-Indexed Reflection submission
- Course wrap-up: "What can you do now that you couldn't in Week 1?"
- Transfer discussion: how these skills apply in degree programs

**In-class activity:** Final submissions. Brief (2-3 min) informal sharing: each student shows one thing they built and says one thing they learned. Not graded; celebratory.

**Language focus:** Reflective language ("I learned...", "I can now...", "What surprised me was...")

**Assessment:**
- Final Artefact due (10%)
- Git Episodes due (10%)
- Evidence-Indexed Reflection due (10%)

## Recurring Weekly Elements

These happen most weeks and are not listed separately above:

| Element | Frequency | Purpose |
|---------|-----------|---------|
| Decision Log entry | ~1 per week (8 total, student-selected) | Ongoing documentation of AI interaction and evaluation |
| Commit to portfolio repository | Weekly | Build Git fluency through repetition |
| Vocabulary check | Weekly | 3-5 new terms introduced; cumulative list maintained |
| AI interaction cycle | Every class | Articulate → review → decide (the core loop) |

## Fluency Development Strategy (Nation's Four Strands)

To address the concern from the design notes about balancing variety and repetition:

| Strand | Where it happens |
|--------|-----------------|
| **Meaning-focused input** | Reading AI output, error messages, technical instructions (every week) |
| **Meaning-focused output** | Decision Logs, verification notes, help requests, PR descriptions (every week) |
| **Language-focused learning** | Vocabulary lists, genre templates, error message parsing (Weeks 2, 5, 7 especially) |
| **Fluency development** | The AI interaction cycle (articulate → review → decide) is the same task type repeated across 12 weeks with different content. Micro-defences repeat the same oral genre 3 times. Decision Logs repeat the same written genre 8 times. |

The key insight: **the task types recur even as the content changes.** Students write 8 Decision Log entries using the same template but about different tools and problems. They do 3 oral walkthroughs using the same structure but about different artefacts. This repetition of language patterns with varying content is exactly what builds fluency.
