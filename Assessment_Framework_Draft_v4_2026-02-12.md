# GEAP 103 Assessment Framework (Draft v4)

**Course:** GEAP 103 Basic Computer Skills
**Version:** Draft v4, 2026-02-12

---

## Grading

- Percentage grades (Humber standard)
- Pass threshold: 50%
- Language level: Exit target B1.2 (IELTS 4.5 / TOEFL PBT 487 / TOEFL iBT 56)
- Entry level: A2+ to low-B1
- Class size: 15-24 students
- Environment: Office 365 + Copilot (baseline; students may use additional AI tools as capabilities evolve)

---

## Assessment Overview

| Assessment | Weight | Timing | Primary CLOs |
|------------|--------|--------|--------------|
| AI Decision Logs | 30% | Ongoing (8 entries, Weeks 2-13) | 1, 4 |
| Oral Micro-Defences + Written Artifacts | 25% | Weeks 4, 8, 12 | 5, 6 |
| Artifact Packages | 25% | Weeks 4, 10, 14 | 3, 6, 7 |
| Git Episodes | 10% | Weeks 6-14 | 2, 8 |
| Evidence-Indexed Reflection | 10% | Week 14 | All |

**Total: 100%**

**Weight rationale:** Process (30%) + Language (25%) + Artifact Package (25%) + Collaboration (10%) + Metacognition (10%). The Decision Logs carry the most weight because they are where students practise the course's core skill: evaluating AI output and making justified decisions. "Artifact Package" means the product together with the documentation, explanation, and sharing that surround it -- the communicative package, not just the thing itself. Git is reduced to reflect the realistic scope for this population in 14 weeks.

## Assessment Details

### 1. AI Decision Logs (30%)

**What:** Structured documentation of AI-assisted work, curated by student, with self-assessment.

**Format:** Template-based log entries (max 8 across semester, student-selected as best evidence):

| Field | Content |
|-------|---------|
| Task goal + constraints | 1-2 sentences |
| AI used | Which AI tool and context (e.g., Copilot in Word, Claude, a coding agent, etc.) |
| Prompt snippet(s) | Max 3 turns (copy/paste or screenshot) |
| Output summary | What the AI proposed (1-2 sentences) |
| Decision | Accept / Modify / Reject |
| Verification | How they checked (ran it, used Accessibility Checker, tested on sample, etc.) |
| What changed | 1-2 bullets |
| **Self-assessment** | **Student rates own entry against rubric (Excellent / Proficient / Developing / Insufficient) + 1 sentence justifying the rating** |

**Evidence cap:** ~150-200 words student-authored text per entry (excluding pasted AI excerpt and self-assessment sentence).

**Failure/rejection episodes:** Entries documenting cases where AI output was wrong or misleading are encouraged throughout. At least one such entry is expected across the 8 submissions, but all entries that document critical evaluation (including rejection or substantial revision of AI output) are valued. Quality of the evaluation matters more than the number of failure episodes.

**CLO Alignment:**
- CLO 1: Task goal articulation, prompt revision
- CLO 4: Evaluation of AI outputs, decision documentation

**Grading Criteria:**

| Level | Description | % Range |
|-------|-------------|---------|
| Excellent | Clear and progressively ambitious goals across entries; systematic evaluation; substantive decisions with rationale; evidence of both exploration (trying new approaches) and exploitation (going deeper on what works); failure/rejection episodes present; self-assessments are well calibrated | 85-100 |
| Proficient | Goals stated with some ambition; evaluation present; decisions documented; entries show range of tasks attempted (not just 8 entries about the same trivial task); at least one failure/rejection episode; self-assessments present | 70-84 |
| Developing | Goals vague or consistently trivial; evaluation superficial; minimal decision rationale; little evidence of exploration or growth in ambition; self-assessments missing or poorly calibrated | 50-69 |
| Insufficient | Missing elements, no critical evaluation visible, no self-assessment | 0-49 |

*Note on ambition: "Ambitious" is calibrated to B1 -- it means the student attempted something that genuinely challenged their ability to articulate what they wanted, not something technically sophisticated. A student who struggles to describe a complex goal and documents that struggle is demonstrating more than a student who describes a trivial goal perfectly. The language is B1; the ambition can be high.*

**Feedback cycle:** Instructor provides written feedback on entries 1-2 (submitted by ~Week 4) before entry 3 is due. Feedback highlights one strength and one specific improvement target. Students are expected to identify in a subsequent entry what they changed based on feedback. A second feedback round covers entries 3-5 (by ~Week 8). Entries 6-8 are assessed summatively.

### 2. Oral Micro-Defences + Written Artifacts (25%)

**What:** Three check-ins across semester combining written technical communication, oral demonstration of understanding, and peer feedback.

**Schedule:** Weeks 4, 8, 12

**Process:**
1. **In-class written diagnostic** (10 min, unmediated): Brief writing task completed in class without AI assistance (e.g., "Describe a problem you encountered this week and how you resolved it" or "Write a help request for [scenario]"). Handwritten or on a locked-down device.
2. Student submits 2 written artifacts before each check-in (e.g., help request + troubleshooting log, or GitHub issue + collaboration note). These may be AI-assisted.
3. 3-4 minute oral walkthrough of ONE artifact
4. Instructor asks 1-2 follow-up questions
5. **Peer feedback:** One peer provides structured feedback on the walkthrough using the task checklist (see below), noting one strength and one question

**Written artifact types (CLO 6):** These are genres of technical communication, not tied to specific tools:
- Help request
- Troubleshooting log
- Bug report
- Issue description (e.g., GitHub Issue)
- Change description (e.g., PR description)
- Collaboration note / handoff documentation
- Process documentation from an artifact package

**Scoring (per check-in):**

**Oral fluency component (30%):**

| Band | Description | % Range |
|------|-------------|---------|
| Above target | Relative ease, able to keep going, pauses for planning evident but doesn't lose thread | 75-84 |
| Target | Can keep going in a way people understand, pauses regularly to think | 65-74 |
| Approaching target | Can say longer things but slow, sometimes has to give up | 50-64 |
| Below threshold | Very short contributions, mostly words/phrases | 0-49 |

*Note: These bands describe oral performance on course tasks. They are not CEFR proficiency ratings. See "B1.2 Exit Evidence" section for how oral performance contributes to the exit proficiency judgment.*

**Task component (50%)** -- checklist:

| Criterion | Evidence |
|-----------|----------|
| Clear problem statement + context | ✓ / partial / ✗ |
| Correct use of essential terminology (5-10 course-defined terms) | ✓ / partial / ✗ |
| Shows what was tried (not just "it doesn't work") | ✓ / partial / ✗ |
| Interprets instruction/error message accurately (or notices uncertainty) | ✓ / partial / ✗ |
| Justifies chosen solution | ✓ / partial / ✗ |

**In-class written diagnostic (10%):** Assessed holistically on communicative effectiveness. This component provides unmediated written evidence of the student's independent language use.

**Peer feedback (10%):** Assessed on whether the peer's feedback is specific, uses course terminology, and identifies a genuine strength or question. (The purpose is to develop the peer's evaluative judgment, not to grade the presenter.)

**CLO Alignment:**
- CLO 5: Interpret technical instructions, troubleshoot, justify solutions
- CLO 6: Communicate technical information clearly in professional genres

**Time requirement:** ~7 min/student (including diagnostic and peer feedback) x 24 students x 3 rounds. Diagnostic is whole-class simultaneous (10 min); oral walkthroughs + peer feedback during lab time while others work independently.

### 3. Artifact Packages (25%)

**What:** Three artifact packages of increasing complexity. Each package includes the product itself AND the documentation, explanation, and sharing that surround it. The artifact is the communicative package, not just the thing.

**An artifact package includes:**
- The product (a digital artifact: document, tool, visualization, application, or whatever the student built)
- Process documentation in the student's own words: what AI did, what the student changed and why, what was checked
- Evidence of sharing or explaining: how the student presented, described, or taught the work to someone else

**Submissions:**

| Package | Week | Weight | Requirements |
|---------|------|--------|--------------|
| Organised file system | 4 | 7% | Logical folder structure, consistent naming, version awareness |
| Mid-semester artifact | 10 | 8% | A working artifact addressing a genuine need, with accessibility where applicable, accompanied by process documentation and evidence of sharing |
| Final artifact | 14 | 10% | More complex or ambitious artifact, iteration visible, documentation shows how the work was explained to others |

*The artifact types are not constrained to Office 365 outputs. Students may build documents, spreadsheets, presentations, visualizations, apps, interactive tools, data pipelines, or whatever serves their goal. The assessment is on the quality of the communicative package: can the student articulate what they built, why, how, and what they learned?*

**Process documentation (required for each package):** Written in the student's own words (not AI-generated):
- What AI did and what the student directed
- What the student changed and why
- What the student checked or verified
- What was shared with others and how (peer review, presentation, explanation)

*The process documentation is a window into the student's independent language use and understanding. Generic or clearly AI-generated notes (e.g., fluent prose markedly above the student's demonstrated oral level) will be flagged for oral follow-up.*

**Accessibility:** Where the artifact is a document, presentation, or other content intended for an audience, accessibility features (headings, alt text, accessible colour, etc.) are expected. The specific tools used to check accessibility may vary; the principle is that the student considered their audience.

**CLO Alignment:**
- CLO 3: Organise, store, retrieve digital files
- CLO 4: Evaluate AI outputs (via process documentation)
- CLO 7: Create and revise digital artifacts to college standards

**Grading Criteria:**

| Level | Description | % Range |
|-------|-------------|---------|
| Excellent | Ambitious attempt; product is functional and accessible; process documentation shows thoughtful AI collaboration in student's own language; evidence of sharing/explaining the work to others; student can articulate what they built, why, and what they learned | 85-100 |
| Proficient | Substantive attempt; product is functional and mostly accessible; process documentation present and in own words; some evidence of sharing | 70-84 |
| Developing | Safe or trivial attempt; product functional with issues; process documentation incomplete or possibly AI-generated; limited evidence of sharing | 50-69 |
| Insufficient | Minimal attempt; non-functional product; missing process documentation | 0-49 |

*Note on ambition: "Ambitious" and "substantive" are calibrated to B1. A student who attempts something that stretches their ability to describe it is being ambitious. A student who sticks to what they can already articulate easily is playing it safe. The framework values the stretch.*

### 4. Git Episodes (10%)

**What:** Evidence of specific Git/version control competencies through required episodes, not quantity metrics.

**Required episodes (each demonstrated once, Weeks 6-14):**

| Episode | Evidence Required | CLO |
|---------|-------------------|-----|
| **Restore** | Demonstrate recovering an earlier version (screenshot/log of revert/checkout/restore) | 2 |
| **Error recovery** | One "something broke → diagnosis → fix" sequence with error text captured | 2, 5 |
| **Collaboration** | One PR (or equivalent) with description + one comment exchange | 8 |

*Conflict Resolution was included in v2 but removed after review. Merge conflict resolution requires a conceptual model of branching and state that is unrealistic for this population in 14 weeks. If authentic conflicts arise during collaborative work, they can be documented as bonus evidence, but they are not required.*

**Assessment:** Checklist-based. Short oral confirmation for any suspiciously "perfect" artifact.

**Target mental model:** By Week 14, students should understand version history as a timeline of saved states that can be navigated (not just "undo"). They should be able to explain, in simple terms, that a commit is a snapshot, that restoring means going back to an earlier snapshot, and that collaboration means two people contributing snapshots to the same timeline. The command-line details are secondary to this conceptual understanding.

**CLO Alignment:**
- CLO 2: Use AI-assisted Git/GitHub workflows
- CLO 8: Collaborate using shared workflows

**Grading:**

| Level | Description | % Range |
|-------|-------------|---------|
| Complete | All 3 episodes documented with clear evidence | 85-100 |
| Mostly complete | 2 episodes documented clearly, 1 weak or missing | 70-84 |
| Partial | 1 episode documented clearly | 50-69 |
| Insufficient | No episodes with clear evidence | 0-49 |

### 5. Evidence-Indexed Reflection (10%)

**What:** Structured reflection linking claims to evidence with evaluative judgment, not open-ended prose.

**Format:** For each of the 8 CLOs:
- 1 claim sentence ("I can now...")
- Links to 1-2 portfolio artifacts as evidence
- 1 sentence evaluating the strength of that evidence ("This is strong/moderate/weak evidence because...")
- 1-2 sentences on what the student would still need to work on or do differently

**Total length:** ~600-800 words (roughly 75-100 words per CLO)

**CLO Alignment:** All (metacognitive synthesis and evaluative judgment)

**Grading Criteria:**

| Level | Description | % Range |
|-------|-------------|---------|
| Excellent | Specific claims, appropriate evidence links, honest evaluation of evidence strength, genuine identification of areas for growth | 85-100 |
| Proficient | Claims present, evidence linked, evidence evaluation attempted, some identification of growth areas | 70-84 |
| Developing | General claims, weak evidence links, no evaluation of evidence quality, vague growth areas | 50-69 |
| Insufficient | Missing CLOs, no evidence links, no evaluative or reflective content | 0-49 |

## Feedback Architecture

Ongoing assessments function as learning progressions only if feedback is timely, specific, and acted upon.

### Decision Logs
- **Round 1** (entries 1-2): Instructor feedback returned by Week 6. One strength, one specific improvement target.
- **Round 2** (entries 3-5): Instructor feedback returned by Week 10. Focus on whether Round 1 feedback was incorporated.
- **Entries 6-8:** Assessed summatively.
- **Self-assessment calibration:** Instructor notes where student self-ratings diverge significantly from instructor assessment. Patterns of over- or under-rating are discussed at the next micro-defence check-in.

### Oral Micro-Defences
- **Peer feedback** at each check-in gives the presenter immediate, structured input from a classmate.
- **Instructor feedback** is oral and immediate (during the check-in). Written scores are returned within one week.
- **Progression signal:** The same task checklist is used at all three check-ins. Students can track their own improvement across the five criteria.

### Artifact Packages
- **Week 4 package** receives written feedback by Week 6, explicitly identifying what to carry forward (in process documentation quality and ambition, not just product quality).
- **Week 10 package** receives written feedback by Week 12, with attention to whether the student's ambition has grown and whether the communicative package (documentation, sharing) is developing.
- **Week 14 package** is assessed summatively.

## Assessor Guidance

### CLO 5 vs CLO 6 Distinction

For marking reliability:

- **CLO 5** (Interpret/troubleshoot): Evidence centres on interpretation and choice under uncertainty. Look for: reading system output, diagnosing problems, selecting a fix, justifying the choice.

- **CLO 6** (Communicate): Evidence centres on packaging information for another person. Look for: genre moves (context, reproduction steps, expected vs actual), polite tone, concise terminology, would-this-work-if-sent-to-someone test.

A single artifact can support both. Don't collapse them into "they explained something."

### Fluency Rubric Usage

Use fluency rubric only for delivery/flow slice of oral evidence. The "target" band represents the expected performance level for B1.2; "above target" is above expectation for this level. Do not report fluency bands as CEFR levels to students or in institutional records. See "B1.2 Exit Evidence" below.

### Failure Episode Verification

Entries documenting failure/rejection of AI output should show:
1. What the AI got wrong
2. How the student noticed
3. What they did instead

Generic or vague entries ("the AI made a mistake so I fixed it") should be flagged for revision at the next feedback round. The goal is authentic critical evaluation, not a compliance checkbox.

### Process Documentation Authenticity

Compare the language in process documentation against the student's demonstrated oral level at micro-defences. A marked mismatch (e.g., fluent academic prose from a student who struggles to sustain oral turns) warrants a brief oral follow-up: "Can you tell me in your own words what the AI did here?"

### Peer Feedback Quality

Peer feedback is assessed on the *peer's* ability to apply criteria, not on the accuracy of their judgment. Look for: specific reference to the checklist, use of course terminology, a genuine (not generic) observation. "Good job" is insufficient; "You explained what you tried before asking for help, which made the problem clear" is sufficient.

## B1.2 Exit Evidence

The B1.2 exit target is a program-level claim supported by converging evidence across the portfolio, not by any single assessment. The evidence base includes:

| Evidence source | Type | What it shows |
|-----------------|------|---------------|
| Oral micro-defences (x3) | Unmediated spoken | Sustained oral communication about technical topics; progression visible across three check-ins |
| In-class written diagnostics (x3) | Unmediated written | Independent written communication without AI assistance |
| Process documentation (x3) | Semi-independent written | Student's own language describing AI collaboration decisions and explaining/sharing their work |
| Decision Log self-assessments (x8) | Semi-independent written | Student's own evaluative language applied to their work |
| Peer feedback (x3) | Semi-independent written | Student's own evaluative language applied to a peer's work |
| Written artifacts (x6) | AI-assisted | Genre competence in technical communication (with AI support) |

**What the framework does not claim:** That any single percentage score on any single assessment corresponds to a CEFR level. The fluency rubric bands are course-internal performance descriptors, not CEFR ratings.

**What the framework does claim:** That a student who passes the course (50%+) has demonstrated, through multiple unmediated and semi-independent performances across 14 weeks, communicative competence consistent with B1.2 in the domain of technical/digital communication. The strongest evidence comes from the unmediated sources (oral defences and in-class diagnostics); the AI-assisted sources provide supporting evidence of genre competence and task articulation.

**Limitation:** This framework assesses language use in a specific domain (digital/technical communication). It does not constitute a general English proficiency assessment. The B1.2 exit target applies to this domain; students' general English proficiency may differ.

## Notes

- Assessment weights: Process (30%) + Language (25%) + Artifact Package (25%) + Collaboration (10%) + Metacognition (10%)
- Early assessments (Weeks 2-6) pitched at A2+; later assessments scaffold toward B1.2
- Git introduced Week 3, assessed from Week 6 (3 weeks practice buffer)
- Oral micro-defences and in-class diagnostics provide unmediated language evidence; written artifacts and decision logs are expected to be AI-assisted (with process documentation and self-assessments in student's own words)
- Accessibility checking tools (e.g., Office 365 Accessibility Checker) treated as authentic workplace tools; specific tools may evolve
- Self- and peer assessment are integrated into existing components (not standalone assessments) to develop evaluative judgment without adding separate graded tasks
- The framework is designed to be tool-independent. AI capabilities will change between the time this framework is written and the course launch. The skills assessed (articulation, evaluation, documentation, communication, collaboration, metacognition) are durable regardless of which tools students use
- Ambition is calibrated to B1: the student attempted something that genuinely challenged their ability to articulate what they wanted. The language level is B1; the project ambition can be high
